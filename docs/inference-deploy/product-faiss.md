## 摘要
FAISS（Facebook AI Similarity Search）是 Facebook 开发的高性能相似性搜索向量库，专注高速向量检索，支持多种索引类型，通过 IndexIDMap 实现向量与元数据的解耦管理。

## 详情

### 基本定位
- **开发方**：Facebook 
- **特点**：**专注高性能相似性搜索**，**检索速度快，支持多种索引类型** 
- **局限性**：**主要用于静态数据，更新/删除复杂** 

### 与其他向量数据库对比
| 数据库 | 特点 | 优势 | 局限性 |
|--------|------|------|--------|
| **FAISS** | Facebook 开发，专注高性能相似性搜索 | 检索速度快，支持多种索引类型 | 主要用于静态数据，更新/删除复杂  |
| **Elasticsearch** | 分布式搜索和分析引擎 | 混合搜索（关键词 + 向量） | 向量检索性能弱于专用数据库  |
| **Milvus** | 开源，支持分布式架构 | 扩展性强，数据管理灵活 | 需要运维能力  |
| **Pinecone** | 托管的云原生向量数据库 | 完全托管，易于部署 | 闭源，成本较高  |
| **Weaviate** | 开源，内置数据向量化模块 | 自动向量化，简化开发流程 | 性能良好但非极致  |
| **Qdrant** | 开源，Rust 开发 | 过滤查询能力强，内存效率高 | 社区相对较小  |

### IndexIDMap 机制
- FAISS 本身**只存储和检索向量，不存储元数据** 
- 使用 `IndexIDMap` 为每个向量指定自定义的 **64 位整数 ID** 
- 用这个 ID 作为元数据存储的键，实现架构解耦 
- 架构：**FAISS（高速向量检索） ←→ 元数据存储（Redis/PostgreSQL/MongoDB）** 

### 基础索引类型
- `IndexFlatL2`：使用 **L2 距离（欧氏距离）进行精确搜索** 
- 用 `IndexIDMap` 包装基础索引，添加带有自定义 ID 的向量 

### 元数据管理方案
| 方案 | 特点 | 适用场景 |
|------|------|---------|
| **Python 列表** | 简单直接，适合原型 | 小规模数据、测试  |
| **Redis（键值数据库）** | 通过 ID 快速查询，性能极高 | 高并发、实时查询  |
| **PostgreSQL（关系型数据库）** | 存储复杂结构化元数据 | 需要复杂查询、事务支持  |
| **MongoDB（文档数据库）** | 存储 JSON 格式元数据 | 灵活 schema、文档型数据  |

### 三步导入法
1. **数据清洗与准备**：确保原始数据质量，进行必要的预处理 
2. **数据向量化（Embedding）**：使用预训练的 Embedding Model 将原始数据转换成向量 
3. **数据与元数据一同导入**：向量 + 唯一 ID + 元数据 

### 核心价值
- **为大模型提供长期记忆**：弥补 LLM 上下文窗口长度限制，解决知识更新延迟问题 
- **实现私有知识库的问答与搜索**：将企业内部文档、产品信息转化为向量，实现基于语义的智能检索 
- **赋能推荐系统、以图搜图等多种应用**：计算用户、物品的向量相似度，提供更精准的推荐 

### 易错点
1. **FAISS 不是数据库**：FAISS 是算法库，不存储元数据 
2. **余弦相似度 vs 欧氏距离**：余弦相似度衡量方向，欧氏距离衡量绝对距离
