# OpenClaw 与 Hermes 智能体系统

## 摘要

OpenClaw 是一个将远端消息入口、会话管理、工具执行回路和长期记忆沉淀缝合在一起的任务系统。Hermes 在 OpenClaw 基础上进行七大核心升级（四级分层记忆、心跳增强、Skill 增强、AI Coding 增强、任务并行、任务失败隔离、核心提示词优化），让智能体从"能做事"进化到"管家级"水平。

## 详情

### 一、OpenClaw 四大核心能力

| 编号 | 能力 | 说明 |
|------|------|------|
| 01 | 远端消息交互 | WhatsApp / Telegram / Slack / 飞书 / 微信 ... |
| 02 | 智能执行 | 问答、总结、检索、生成、工具调用 |
| 03 | 持续协作 | Session、上下文、长期记忆、后续跟进 |
| 04 | 可操作 | 浏览器、系统工具、消息动作、设备能力 |



**总结**：OpenClaw 更像一个能被远程触发、能持续协作、还能动动手做事的**任务入口层** 。

### 二、模型应用形态的演进

```
ChatGPT (Prompt/问答/单轮推理) → GPTs (预设/工具调用/轻知识库) → 低代码 (工作流/可视化/去IDE) → Coding Agent/CLI (系统环境/复杂任务/本地命令) → OpenClaw (远程交互/长时运行/长期记忆)
```



**关键观察**：OpenClaw 并没有在"智能"本身上产生决定性跃迁，但它第一次把"**远程消息 IO + 会话 + 工具 + 回写 + 记忆**"的长时运行任务系统直观做给了用户看 。

### 三、核心响应过程

主链路：**入口 → 接入 → 准备 → 主回路 → 回写 → 沉淀 → 继续 / 结束** 。

**三大特性**：
| 特性 | 含义 |
|------|------|
| 可见 | 过程里发生什么，渠道和 UI 能持续看到 |
| 可恢复 | 超时、重试、压缩上下文、模型切换都在同一条链里 |
| 可继续 | 这一轮做过的事，会变成下一轮还能接着用的状态 |



### 四、关键构件分层

| 层级 | 构件 | 说明 |
|------|------|------|
| 会话控制层 | Gateway | 总线 / 鉴权 / 连接 / 事件 |
| | Session | 上下文桶 / 路由落点 / 并发边界 |
| 智能层 | Agent Loop | 一轮一轮推进的模型 + 工具执行回路 |
| | Memory | 落在外部存储里的长期记忆 |
| 行动能力层 | Skills | 教会 agent 在什么场景怎么用工具 |
| | ACP | 利用 ACPX 与其他 Agent 协作完成任务 |



**可扩展性总结**：
- Gateway 高（天然适合横向扩展更多接入和 IO 能力）
- Session 低（分桶策略不宜随意改动）
- Memory 低（长期记忆主要沉淀在外部存储）
- Agent Loop 低（统一、可观察，但 Token 耗用大）
- Skills 高（更像是在外层补行为策略）
- ACP 高（把真正的任务处理交给其他 Agent）

### 五、Memory（记忆机制）

OpenClaw 的记忆以 Markdown 文件形式存储，结构清晰、人人可读写 。

**记忆文件体系**：
| 文件/目录 | 作用 | 说明 |
|-----------|------|------|
| `memory/YYYY-MM-DD.md` | 每日记录 | 记录当日发生的原始日志 |
| `MEMORY.md` | 长期记忆 | 精选记忆库，如同人类的长期记忆 |



**核心特性**：可随时查看记忆文件；如果认为 AI 记错了，可人工修改记忆文件内容；做任务时自动加载今日 + 昨日的 .md 文件；通过关键词搜索、向量搜索，检索任务相关的记忆 。

**MEMORY.md 详细说明**：
- 加载时机：仅在主会话加载（与使用者直接对话时）
- 禁止加载场景：共享场景禁止加载（Discord、群聊、与他人会话时）
- 安全考虑：其中包含私密信息，不可泄露给陌生人 
- 记录内容：重要事件、思考、决策、观点、经验教训 
- 性质：精炼后的核心记忆，而非原始日志 

**关键原则**：务必落字为记，勿存"脑中备忘"。记忆容量有限，想记住的内容必须写入文件。"脑中想法"不会在会话重启后保留，文件才会 。

**记忆存储策略**：LanceDB 优先，本地文件为降级方案 。

### 六、心跳（Heartbeat）机制

心跳轮询约 20-40 分钟一次，读取 HEARTBEAT.md 任务清单，批量检查邮件/日历/通知等 。

**心跳 vs 定时任务（Cron）**：
| 维度 | 心跳（Heartbeat） | 定时任务（Cron） |
|------|-------------------|------------------|
| 适用场景 | 多项检查可批量执行（收件箱 + 日程 + 通知一次完成） | 需要精准时间（如"每周一早上 9 点整"） |
| 上下文需求 | 需要近期消息的对话上下文 | 任务需与主会话历史隔离 |
| 时间要求 | 时间可略有浮动（约 30 分钟一次即可，无需精准） | 需要不同模型或思考强度的任务 |



**每日巡检内容**：邮件（是否有紧急未读消息？）、日程（未来 24-48 小时是否有即将到来的事项？）、社交提及、天气 。

**保持静默场景（HEARTBEAT_OK）**：深夜时段 23:00 - 08:00 非紧急情况；使用者明显处于忙碌状态；上次巡检后无新内容；距上次巡检不足 30 分钟；无需请示可主动完成的工作 。

### 七、Skill（技能体系）

**核心特性**：纯文本定义（人人可写，易维护、可审计、可版本管理）；三级渐进式加载（元数据加载 → 全文加载 → 执行加载）；多层级来源（工作区技能、全局技能、内置技能、插件技能）；双触发模式（自然语言描述、直接命令调用）；沙箱隔离与权限管控 。

### 八、核心提示词文件体系

| 文件 | 作用 | 说明 |
|------|------|------|
| AGENTS.md | 工作空间规则指南 | 此文件夹即为你的专属领地，请按此规则行事 |
| SOUL.md | 安全规则和做人准则 | 核心身份、人格与安全边界 |
| IDENTITY.md | 身份信息 | 我的身份信息 |
| USER.md | 服务对象信息 | 你的信息（偏好、沟通风格等） |
| TOOLS.md | 本地工具配置笔记 | 包含 Coze API 配置、魔法移除说明、PPT 生成等 |
| HEARTBEAT.md | 心跳任务清单 | 心跳轮询时执行的任务清单 |
| BOOTSTRAP.md | 初始设定文件 | 首次运行时存在，完成后删除，此后不再需要 |



**每次会话开始**：务必阅读 SOUL.md → USER.md → memory/YYYY-MM-DD.md（今日与昨日文件）→ 若处于主会话额外阅读 MEMORY.md → 无需请示，直接执行即可 。

### 九、Hermes 核心升级

**核心判断**：Hermes 通过四级分层记忆、心跳增强、Skill 增强、AI coding 增强、任务并行、任务失败隔离等核心升级，让智能体从"能做事"进化到"管家级"水平，从"单任务串行"进化到"多任务并行"，从"主 Agent 会挂"进化到"主 Agent 永不挂" 。

### 十、Hermes 四级分层记忆

| Tier | 名称 | 内容来源 | 说明 |
|------|------|----------|------|
| Tier 0 | 标题/核心关键词 | 从 Tier 1 摘要里自动提取，或从原始文件自动提炼标题 | 仅标题/核心关键词，最精简 |
| Tier 1 | 摘要 | 对原始全文（Tier 2）做压缩总结 | 原文 2000 字 → 压缩成 100~200 字要点 |
| Tier 2 | 完整内容 | 当天完整对话日志，完整聊天记录 | 最原始的内容本体 |
| Tier 3 | 深层参考 | 自动关联、自动抓取的相关历史记忆、技能文档、代码片段、API 文档等 | 扩展资料，提供深度上下文 |



**存储方案**：所有聊天记录、会话历史、结构化状态 → 全进 SQLite。优先 LanceDB，也可以降级到 `memory/YYYY-MM-DD.md` + `MEMORY.md` 。

### 十一、OpenClaw vs Hermes 完整对比

| 维度 | OpenClaw | Hermes |
|------|----------|--------|
| 记忆结构 | 两级（每日记录 + 长期记忆） | 四级（标题→摘要→全文→深层参考） |
| 状态持久化 | 内存存储，重启丢失 | JSON 持久记录 |
| 心跳频率 | 固定间隔 | 动态智能调整 |
| 主动工作 | 简单任务 | 管家级维护 |
| 记忆联动 | 弱关联 | 心跳 = 记忆维护核心时机 |
| Skill 创建 | 半自动 + 人工引导 | 完全静默、全自动 |
| Skill 加载 | 三级渐进式加载（元数据→全文→执行） | 三级渐进式加载（Level 0/1/2，token 精确控制） |
| Skill 进化 | 外挂技能 | 内置 GEPA 进化算法 |
| 项目记忆 | 依赖 .md 文件 | LanceDB 向量记忆 |
| 调试能力 | 会卡循环 | 自主反思机制 |
| 长期稳定性 | 偶尔需要重启 | 7×24 小时不崩不卡 |
| 任务执行 | 单任务串行 | 多任务并行 |
| 容错能力 | 主 Agent 可能受影响 | 子 Agent 失败，主 Agent 永不挂 |



### 十二、任务并行与失败隔离

**任务并行**：主 Agent 聊天和调度，同时派生出多个子 Agent 干活。子代理一无所知，这才是关键。主 Agent 只传递：当前要干的任务（必传）、项目上下文（精简版）、与任务直接相关的对话片段、权限与环境信息 。

**任务失败隔离**：重型任务永远子 Agent 干活，主 Agent 永远不会挂。子 Agent 失败不影响主 Agent 和其他子 Agent，用户始终能与主 Agent 正常对话 。

### 十三、准确率与风险矩阵

| | 低行动风险 | 高行动风险 |
|---|---|---|
| 高正确率 | ✅ 政策/制度问答草稿，带证据的知识检索，告警摘要/分诊 | ⚠️ 正式批复/审批/财税/执法，自动改写核心业务数据 |
| 低正确率 | 🟡 内容草稿/个人简报/会议纪要，个人助理/小团队提醒 | ❌ 自动发客户消息，高权限浏览器/内网动作，读写生产系统 |



**系统工程公式**：任务约束 + 预算约束 + 验证约束 + 泄露约束 。

### 十四、OpenClaw 定位

OpenClaw 更适合作为**入口层、网关层、协作层**；越往复杂任务内核走，越需要更强的 workflow 和 loop 。

| 类型 | 适用场景 | 优势/强项 | 短板 |
|------|----------|-----------|------|
| 单体 ReAct 智能体 | 个人任务、轻自动化、内容草稿 | 简单、快、产品感强 | 复杂任务容易靠模型硬扛，纠偏与验证弱 |
| OpenClaw 路由中枢 | 把消息入口、会话、权限、动作统一收口 | 擅长处理从哪进、回哪去、带什么上下文、该调用什么/谁来处理 | 依然是 ReAct 的简单回路 |
| Agentic Workflow 系统 | 高正确率、长任务、多阶段验证 | 规划-执行-校验-审批-行动等标准行动流 | 需要更深入研究智能系统本身，对技术能力要求高 |
