# Harness 四问法（边界判断工具）

> **类型**: concept
> **创建时间**: 2026-07-05
> **最后更新**: 2026-07-05
> **来源**:  第 5 章
> **领域**: ai

## 摘要

四问法来自 Sanderson Oliveira de Macedo 2026 预印本 *What makes a harness a harness*（[arXiv 2606.10106](https://arxiv.org/abs/2606.10106)），是一种**边界判断工具**而非行业标准。判断一个对象是不是完整 Harness，可以不问名字，而是问 4 个运行时问题：T1 闭环、T2 环境改变、T3 任务感知上下文、T4 独立控制。**四项都"是"时，才适合称为完整 Harness**。

## 详情

> 🖼️ [图 5] **四问法** — 判断一个对象是不是完整 Agent Harness 的 4 个并列问题：(1) 有没有运行时循环？观察结果会不会改变下一步；(2) 能不能作用于环境？工具能读取并改变外部状态；(3) 有没有任务感知的上下文/状态？不是把全部聊天记录原样推回去；(4) 有没有独立于模型服从的控制？权限、测试、策略或状态检查 — 

### 一、四问法概述

> 完整 Agent Harness：**它包在一个或多个模型周围，用行动循环、环境工具、上下文管理和独立控制，把模型变成能在外部环境中完成任务的 Agent。**

四问法适合做**边界测试**，**不适合当功能优先级清单**——一个系统不完整不代表它没用，它可能只是某个关键组件。

### 二、4 个问题详解

#### T1：运行时是否存在"推理/决策—行动—观察"闭环？

| 维度 | 内容 |
|------|------|
| **通过条件** | 上一步的环境观察能够改变下一步行动，而不是永远按固定清单执行 |
| **为什么必要** | 没有闭环，系统只是单次生成器或固定流水线，不能根据现实反馈持续推进 |
| **通过示例** | 运行测试后根据失败信息决定修改哪段代码 |
| **不通过示例** | 固定执行"检索→生成→格式化"，观察结果不能改变路径 |

#### T2：模型是否能通过接口感知并改变外部环境？

| 维度 | 内容 |
|------|------|
| **通过条件** | 工具不只返回知识，还允许写文件、运行命令、操作浏览器、调用业务系统 |
| **为什么必要** | Agent 任务的结果存在于外部世界；如果模型只能写建议，它仍是助手而不是执行系统 |
| **通过示例** | 编辑仓库、创建工单、更新数据库（受权限约束）|
| **不通过示例** | 只给出修改建议文本，不实际改变环境 |

#### T3：系统是否主动、按任务内容管理上下文和状态？

| 维度 | 内容 |
|------|------|
| **通过条件** | 进入/离开模型窗口的信息由当前任务和观察决定，而不只是把全部历史堆回去或按长度截断 |
| **为什么必要** | 多步任务会不断产生工具结果和状态；没有选择机制，相关信息会被噪声淹没 |
| **通过示例** | 从 Progress、Plan、当前失败日志中选择下一步所需信息 |
| **不通过示例** | 每轮原样追加全部聊天记录；超长时机械删除最早消息 |

#### T4：是否至少有一种不依赖"模型自觉服从"的控制或验证？

| 维度 | 内容 |
|------|------|
| **通过条件** | 权限、预算、测试、策略门、状态断言等机制能独立阻断、改变或确认执行 |
| **为什么必要** | Prompt 中的"不要做"仍取决于模型听话；系统需要可强制的边界和独立完成证据 |
| **通过示例** | 工具调用上限、只读权限、单元测试、审批门、幂等键 |
| **不通过示例** | 只记录日志，或只在 Prompt 里要求"请勿删除文件" |

### 三、四问法判定矩阵

| 对象 | T1 闭环 | T2 改环境 | T3 上下文 | T4 独立控制 | 严格判定 |
|------|:------:|:------:|:------:|:------:|---------|
| 单次 Prompt 模板 | ❌ | ❌ | ❌ | ❌ | 组件 / 邻接系统 |
| 仅 `AGENTS.md` / `CLAUDE.md` | ❌ | ❌ | ❌ | ❌ | 组件 / 邻接系统 |
| MCP Server / 工具集合 | ❌ | ✅ | ❌ | ❌ | 组件 / 邻接系统 |
| Sandbox / 容器 | ❌ | ✅ | ❌ | ✅ | 组件 / 邻接系统 |
| 固定 DAG 工作流 | ❌ | ✅ | ❌ | ✅ | 组件 / 邻接系统 |
| 仅 `while loop + tools`（无持久状态/验证）| ✅ | ✅ | ❌ | ❌ | 组件 / 邻接系统 |
| **成熟 Coding Agent 产品**（完整配置）| ✅ | ✅ | ✅ | ✅ | **完整 Harness** |
| **Agent + 状态 + Policy + Tests** | ✅ | ✅ | ✅ | ✅ | **完整 Harness** |

### 四、关键警告

> 配套资料警告：
> - **T3 不是"有聊天历史"就算**，而是任务感知的选择
> - **T4 不是"模型说自己检查过"就算**，而是控制效果不依赖模型是否合作
> - **严格口径下，T2 也不只是"能看见环境"**；接口还应允许改变环境

### 五、四问法的核心价值

> 四问法最有用的地方，**不是给对象贴"高级/低级"标签，而是把缺口说清楚**。

- MCP Server、Sandbox、固定 DAG 工作流都可能是关键组件，但**单独看通常不构成完整 Harness**
- 仅有 `while loop + tools` 看起来很像 Agent，**但没有持久状态和独立验证时，长任务很容易把错误一路带下去**
