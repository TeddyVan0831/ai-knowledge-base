# RAG 高级检索技术

> **类型**: concept
> **创建时间**: 2026-05-26
> **最后更新**: 2026-05-26
> **来源**: 

## 摘要

RAG 高级检索技术涵盖混合检索（BM25 + Vector）、重排序（Rerank 模型）、MultiQuery 查询扩展、双向改写、Small-to-Big 索引策略，以及 GraphRAG 知识图谱结构化检索方法，用于提升 RAG 系统的召回精度和全局理解能力。

## 详情

### RAG 调优全景图

RAG 高级技术与调优分为三个层次 :

1. **坚实地基：知识库处理** — 问题生成与检索优化、对话知识沉淀、健康度检查、版本管理与性能比较
2. **精准雷达：高效召回** — MultiQuery & 改写、BM25 + Vector 混合检索、Rerank 精细排序
3. **全局视野：GraphRAG** — 知识图谱构建（实体抽取 & 社区摘要）、Global vs Local 查询模式

### 混合检索：BM25 + Vector

结合关键词检索（离散索引）和语义检索（向量索引）的优势 :

| 检索方法       | 原理              | 优势       | 劣势        | 适用场景          |
| -------------- | ----------------- | ---------- | ----------- | ----------------- |
| **BM25**   | 词频/逆文档频率        | 精确匹配专有名词 | 无法理解语义    | "投诉扣分"精确查找    |
| **Vector** | Embedding 语义相似度 | 理解同义词、语义 | 可能漏掉精确关键词 | "处罚规定"="扣分标准" |

BM25是一种经典的文本检索算法，它是TF-IDF（词频-逆文档频率）的改进版本，通过更精细的词频饱和度和文档长度归一化，提升了检索的相关性排序效果 。

**混合检索流程** :
```
Step 1: 输入 → 用户查询进入系统
Step 2: 并行检索 → BM25（分词→词频匹配）+ 向量检索（Embedding→相似度计算）
Step 3: 归一化 → 两种分数统一缩放到 [0, 1] 区间
Step 4: 加权融合 → Score = α × Vector + (1-α) × BM25
Step 5: 输出 → 按融合分数排序，返回 Top-K 结果
```

**Alpha 权重调优** :

| Alpha 值     | 检索倾向         | 适用场景          |
| ----------- | ------------ | ------------- |
| **α = 0.0** | 纯 BM25，只看关键词 | 专业术语查询、精确匹配需求 |
| **α = 0.3** | 偏向关键词，兼顾语义   | 技术文档、法规条文检索   |
| **α = 0.5** | 平衡，各占一半      | 通用场景（推荐默认值）   |
| **α = 0.7** | 偏向语义，兼顾关键词   | 口语化问答、模糊查询    |
| **α = 1.0** | 纯向量，只看语义     | 同义词丰富、表述多样的场景 |

### 精细排序：Rerank 模型

对初步检索结果进行重排序，提高最终输出的相关性 。

**主流 Rerank 模型对比** :

| 特性        | BGE-Rerank | Cohere Rerank |
| ----------- | ---------- | ------------- |
| **类型**    | 开源         | 商业 API        |
| **部署方式**  | 可本地部署      | 云端调用          |
| **多语言支持** | 中英优化       | 多语言（v3.0）     |
| **适用场景**  | 数据敏感、垂直领域  | 快速集成、多语言优化    |

**BGE-Rerank 分数范围** :
- **高相关性**：3.0~10.0
- **中等相关性**：0.0~3.0
- **低相关性/不相关**：负数（如 -5.0 以下）

**参数设置建议** :

| 参数           | 说明             | 建议值                           |
| ------------ | -------------- | ----------------------------- |
| `initial_k`  | 粗排召回数量         | 10-20 |
| `final_k`    | 最终返回数量         | 3-5 |
| `max_length` | Tokenizer 最大长度 | 512 |
| `alpha`      | 混合检索权重         | 0.5 |

### MultiQuery 查询扩展

使用 LLM 将用户查询改写成多个语义相近的查询，提升召回多样性 。

**示例** :
```
原始查询："客户经理被投诉了，投诉一次扣多少分"
生成的查询变体:
- 客户经理被投诉了，投诉一次扣多少分
- 客户经理投诉扣分标准是什么
- 银行客户经理被投诉一次会扣除多少绩效分
- 金融机构客户经理投诉处罚机制
```

### 双向改写（Doc2Query / Query2Doc）

针对短文本向量化效果差的优化策略，通过文本扩展，补充短文本的语义信息 。

- **Doc2Query**：为短文档/段落生成多个模拟用户可能提出的查询语句，为文档"增加语义锚点" 。
- **Query2Doc**：将短查询/关键词扩写为完整的、包含上下文的文档式文本，弥补信息缺失 。

**工程落地注意事项** :
1. Doc2Query 一般生成 3-5 个查询即可
2. 需过滤重复语义的冗余内容
3. 可使用轻量级大模型降低推理成本
4. 双向改写后可搭配 Rerank 模型进一步提升精度

### Small-to-Big 索引策略

通过小规模内容（摘要、关键句）建立索引，链接到大规模内容主体 。优势在于能够快速定位相关的小规模内容，并通过链接获取更详细的上下文信息。

```
用户查询 → 小规模内容检索 → 链接到大规模内容 → 上下文补充 → 生成答案
```

### GraphRAG

GraphRAG 是一种结构化的、分层的检索增强生成方法，而不是使用纯文本片段的语义搜索方法 。

**GraphRAG vs 基线 RAG** :

| 对比维度     | 基线 RAG    | GraphRAG   |
| ------------ | ----------- | ---------- |
| **检索方式** | 向量相似性搜索   | 知识图谱结构化检索  |
| **连接能力** | 难以连接各个要点  | 通过图谱关系遍历信息 |
| **宏观理解** | 难以概括大文档主旨 | 社区摘要提供全局视角 |
| **适用问题** | 具体事实查询    | 复杂推理、全局性问题 |

**GraphRAG 索引构建流程** :
1. **切片**：将文档切分成文本块（TextUnits）
2. **抽取**：使用 LLM 从文本块中提取实体（人、地、物）、关系、主张
3. **聚类**：使用 Leiden 算法对知识图谱进行层次聚类
4. **摘要**：自下而上为每个社区生成摘要，建立层级汇报体系

**查询模式** :

| 对比维度       | Global 模式               | Local 模式           |
| -------------- | ------------------------- | ------------------- |
| **适用场景**   | 全局型查询任务（概要性问题）          | 具体事实提问（特定实体信息）     |
| **查询架构**   | Map-Reduce 架构           | 知识图谱 + 原始文档结合      |
| **LLM 调用** | 多次调用（每个社区生成答案再汇总）       | 一次调用（上下文 + 问题一起输入） |
| **数据来源**   | 主要基于社区报告                | 实体、关系、社区报告、原始文本块   |
| **查询成本**   | 非常高                     | 高                  |
