## 摘要

RAG 优化涉及多个维度的调优策略，包括 Query 改写、联网搜索集成、分块策略选择、混合检索、重排序和系统调试方法，是构建高质量 RAG 系统的综合性主题。

## 详情

### Query 改写优化

Query 改写是 RAG 检索阶段的前置优化，将口语化、模糊查询转换为精确检索语句。包含五种类型 ：

1. **上下文依赖型** — 包含"还有"、"其他"等需要上下文理解的词汇
2. **对比型** — 包含"哪个"、"比较"、"更"等比较词汇
3. **模糊指代型** — 包含"它"、"他们"、"都"、"这个"等指代词
4. **多意图型** — 包含多个独立问题，需拆分处理
5. **反问型** — 包含"不会"、"难道"等反问语气

自动意图识别时，如果同时存在多意图型、模糊指代型，优先级为**多意图型 > 模糊指代型** 。

### 联网搜索集成

当 query 涉及时效性信息时，需要联网搜索后才能回答。分为三个步骤 ：

1. 识别是否需要联网搜索
2. 为联网搜索改写查询
3. 生成搜索策略

需要联网搜索的场景共 8 类：时效性、价格信息、营业信息、活动信息、天气信息、交通信息、预订信息、实时状态 。

### 分块策略选择

分块策略选择没有最好的，只有最适合场景的 。选择建议 ：

| 场景        | 推荐方法     | 理由   |
| ----------- | ---------- | ------ |
| **通用场景**  | 固定长度切片   | 简单可靠 |
| **技术文档**  | 层次切片     | 保留结构 |
| **高质量要求** | LLM 语义切片 | 效果最好 |
| **长文档召回** | 滑动窗口切片   | 不漏信息 |

分块大小一般推荐 **256-1024 tokens** ，常见配置为 `chunk_size=512, overlap=50-100` 。

### 混合检索调优

混合检索结合 BM25（精确匹配）和 Vector（语义理解），通过 Alpha 权重控制倾向 ：

- 专业术语多的文档 → 降低 alpha（偏 BM25）
- 用户问题口语化 → 提高 alpha（偏向量）
- 不确定时 → 从 α = 0.5 开始，根据效果微调

加权融合公式：`Score = α × Vector + (1-α) × BM25` 。

### 重排序优化

Rerank 模型对每个 (Query, Doc) 对都要过一次，候选数量不宜过多 。

参数建议 ：
- `initial_k`: 10-20（粗排召回数量）
- `final_k`: 3-5（最终返回数量）
- `max_length`: 512（Tokenizer 最大长度）

### RAG 系统调试

按照 RAG 的流程逐步排查问题 ：

**检索阶段调试** :

| 问题现象       | 可能原因                | 解决方案                      |
| -------------- | ----------------------- | --------------------------- |
| 召回内容不相关    | Embedding 模型对领域词理解差 | 换用领域微调的 Embedding，或加同义词扩展 |
| 答案散落在多个块   | 分块太小，信息被切断          | 增大 chunk_size，增加 overlap  |
| 噪声太多       | 分块太大，混入无关内容         | 减小 chunk_size，添加 Rerank   |
| 用户口语化问题检索差 | Query 与文档表述风格差异     | Query改写，HyDE 假设文档生成       |

**生成阶段调试** :

| 问题现象      | 可能原因           | 解决方案                     |
| ------------- | ------------------ | ---------------------------- |
| 答案与检索内容不符 | LLM 幻觉，未遵循上下文  | 强化 Prompt 指令："仅基于背景知识回答" |
| 答案过于简短    | Prompt 未要求详细解释 | 添加输出格式要求                 |
| 答案冗长有很多废话 | 上下文噪声多         | Rerank 精选，减少喂给 LLM 的内容   |
| 无法处理复杂推理  | LLM 能力不足       | 换用更强的模型，或添加 CoT          |

### RAG 系统评估

RAG 系统的评估需要从检索质量和生成质量两个维度进行 。

**检索质量指标** :

| 指标              | 定义               | 计算公式             | 适用场景      |
| ----------------- | ------------------ | -------------------- | ----------- |
| **Recall@K**    | Top-K 中包含相关文档的比例 | 相关文档数 / 总相关文档数   | 评估召回能力    |
| **Precision@K** | Top-K 中相关文档的比例   | 相关文档数 / K        | 评估精准度     |
| **MRR**         | 第一个相关文档的排名倒数     | 1 / (第一个相关文档的排名) | 单一正确答案场景  |
| **NDCG@K**      | 考虑相关性等级的排序质量     | DCG / IDCG       | 有相关性分级的场景 |

**生成质量指标（RAGAS 框架）** :

| 指标                             | 定义                 |
| -------------------------------- | -------------------- |
| **Faithfulness (忠实度)**         | 答案是否基于检索到的内容，而非幻觉  |
| **Answer Relevance (答案相关性)**   | 答案是否回答了用户的问题       |
| **Context Relevance (上下文相关性)** | 检索到的内容是否与问题相关      |
| **Context Recall (上下文召回)**     | 检索是否召回了回答问题所需的所有信息 |

评估建议：构建包含 50-100 个样本的评估集，覆盖各类问题；重点关注 **Faithfulness**，这是 RAG 的核心价值 。
