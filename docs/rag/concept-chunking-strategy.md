# 分块策略（Chunking Strategy）

> **类型**: concept
> **创建时间**: 2026-05-26
> **最后更新**: 2026-05-26
> **来源**: 

## 摘要

分块策略（Chunking）是 RAG 系统的核心环节，直接影响检索质量和回答准确性。本文对比五种常用切片方法：固定长度切片、句子边界切片、LLM 语义切片、层次切片和滑动窗口切片，以及分块大小的经验值建议。

## 详情

知识切片是RAG系统的核心环节，直接影响检索质量和回答准确性 。

### 五种切片方法对比

来源: 

| 方法           | 核心思路            | 重叠  | 长度均匀 | 语义完整 | 实现成本 | 适用场景         | 不适用场景      |
| -------------- | ------------------- | ----- | ------ | ------ | ------ | -------------- | ------------- |
| **固定长度切片**   | 按固定字符数切，句子边界优化  | 有   | 高    | 中    | 低    | 通用场景、批量处理    | 语义敏感的问答场景  |
| **句子边界切片**   | 按句子、段落等语义单位进行切分 | 无   | 低    | 高    | 低    | 自然语言文本、问答系统  | 长句子多的文档    |
| **LLM 语义切片** | LLM 理解内容后切分     | 无   | 中    | 最高   | 高    | 高质量要求、复杂语义   | 大规模文档、成本敏感 |
| **层次切片**     | 按标题/章节切分        | 无   | 低    | 高    | 中    | 结构化文档（手册、规范） | 无标题的纯文本    |
| **滑动窗口切片**   | 固定窗口滑动，大量重叠     | 大量  | 高    | 中    | 低    | 长文档召回、上下文连续  |               |

### 固定长度切片

**特点**：按固定字符数切分，优先在句子边界切分 。

**示例** :
```
块 1 (292 字符): 迪士尼乐园提供多种门票类型...购票渠道以官方渠道为主...
块 2 (80 字符): 需在官方渠道登记...军人优惠现役及退役军人凭证件享 8 折...
```

**适用**：需要统一处理长度的场景，批量处理大量文档 。

### 句子边界切片

**特点**：基于 NLP，按句子、段落等语义单位切分 。

**示例** :
```
块 1 (287 字符): 迪士尼乐园提供多种门票类型...（完整句子）
块 2 (29 字符): 军人优惠现役及退役军人凭证件享 8 折...
```

**适用**：自然语言文本，需要保持语义完整性的场景 。

### LLM 语义切片

**特点**：利用 LLM 语义理解能力，精确控制长度 。

**结果示例** :
```
块 1 (57 字符): 迪士尼乐园提供多种门票类型...价格根据季节浮动。
块 2 (56 字符): 两日票需要连续两天使用...需注意门票标注的有效期限。
块 3 (71 字符): 购票渠道以官方渠道为主...但需认准官方授权标识。
```

**适用**：高质量要求的场景，复杂语义结构，有预算支持 。

### 层次切片

**特点**：基于文档的层次结构（标题、章节、段落）切分 。

**示例** :
```
块 1 (11 字符): # 迪士尼乐园门票指南
块 2 (219 字符): ## 一、门票类型介绍 ### 1. 基础门票类型...
块 3 (214 字符): ## 二、购票渠道与流程 ### 1. 官方购票渠道...
```

**适用**：结构化文档（手册、规范、API 文档）。

### 滑动窗口切片

**特点**：固定大小窗口滑动，产生大量重叠切片 。

**示例** :
```
块 1 (299 字符): 迪士尼乐园提供多种门票类型...军人优惠现役及...
块 2 (173 字符): ...第三方平台如飞猪、携程...军人优惠现役及退役...
块 3 (23 字符): ...退役军人凭证件享 8 折...
```

**适用**：需要上下文连续、长文档召回的场景 。

### 切片策略选择建议

来源: 

| 场景        | 推荐方法     | 理由   |
| ----------- | ---------- | ------ |
| **通用场景**  | 固定长度切片   | 简单可靠 |
| **技术文档**  | 层次切片     | 保留结构 |
| **高质量要求** | LLM 语义切片 | 效果最好 |
| **长文档召回** | 滑动窗口切片   | 不漏信息 |

### 分块大小经验值

根据项目文档实际情况调整 chunks 大小，一般推荐 **256-1024 tokens** 。

- 小块: 检索精度高，但可能丢失上下文 
- 大块: 上下文完整，但噪声多，检索精度下降 
- 常见配置: `chunk_size=512, overlap=50-100`
