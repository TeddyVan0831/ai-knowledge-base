# Embedding 模型选型

> **类型**: concept
> **创建时间**: 2026-05-26
> **最后更新**: 2026-05-26
> **来源**: 、

## 摘要

Embedding 模型选型基于 MTEB 榜单和具体业务场景（检索/分类/聚类），需权衡维度、上下文窗口、推理速度和部署成本，核心原则是"够用就好，维度不是越高越好"。

## 详情

### MTEB 评测基准

MTEB（Massive Text Embedding Benchmark）涵盖 **8 大类任务**  和 **58 个数据集** ：

| 任务 | 说明 | 典型场景 |
|------|------|---------|
| 检索  | 从文档库找出最相关文档 | RAG、搜索引擎 |
| STS  | 判断句子对语义相似程度（1-5 分） | 语义去重、推荐 |
| 重排序  | 对初步检索结果二次排序 | RAG 召回优化 |
| 分类  | 将文本划分到预定义类别 | 文本分类、情感分析 |
| 聚类  | 无标签自动分组 | 主题发现 |
| 对分类  | 判断文本对是否具特定关系 | 重复问题检测 |
| 双语挖掘  | 找出跨语言翻译对 | 机器翻译 |
| 摘要  | 评估生成摘要与参考摘要的语义相似度 | 摘要质量评估 |

### 向量维度对性能的影响

| 维度 | 特点 | 适用场景 |
|------|------|---------|
| **高维度 (1024, 4096)**  | 编码更丰富、语义更细致 | 大规模信息检索、细粒度文本分类 |
| **低维度 (256, 512)**  | 计算速度快，内存占用小 | 移动端、实时性要求高的场景 |

**性价比分析**：
- 从 768 维拉长到 1024 维，检索指标提高不到 1%，但内存多占约 35% → **不需要** 
- 压缩到 768 维后，指标下降超过 5% → **值得使用更高维度** 

### 主流模型对比

| 模型 | 特点 | 文件大小 | 适用场景 |
|------|------|---------|---------|
| **BGE-M3**  | 100+ 语言，8192 tokens，密集+稀疏+多向量 | 2.3G | 跨语言长文档检索 |
| **text-embedding-3-large**  | 3072 维，长文本语义强，英文优秀 | - | 英文全球化应用 |
| **Jina-embeddings-v2-small**  | 35M 参数，RT<50ms，轻量化 | - | 轻量级实时推理 |
| **Qwen3-Embedding-8B**  | MTEB Mean Score 70.58，4096 维，32768 tokens | - | 高精度中文 RAG |

### 选型步骤

1. **明确业务场景与评估指标** 
   - 定义核心任务：检索、分类还是聚类？
   - 确定衡量指标：Recall@K、Accuracy、NDCG
2. **构建"黄金"测试集** 
   - 准备高质量小规模测试集
   - 构建"问题 - 标准答案"对
3. **小范围对比测试** 
   - 从 MTEB 榜单挑选候选模型
   - 使用黄金测试集评测
4. **综合评估** 
   - 结合测试结果、推理速度、部署成本做出决策
